# 民盟工作知识库工程控制面

本仓库用于承载“民盟工作知识库”的工程控制面：规划、流程、脚本、公开样例、交接工单和复核记录。

它不再只是微信公众号语料库，而是面向完整工作资料底座的本地 RAG 参谋系统。核心目标是把长期工作资料、公众号公开语料、上海盟讯 OCR、人物档案 OCR 和后续外部公开研究资源，组织成可检索、可追溯、可复核的工作知识库。

## 项目目标

- 建立个人工作的长期知识底座。
- 支持文稿、盟史、人物、会议、宣传、参政议政等工作场景的快速检索和材料组织。
- 用关键词检索找精确文件，用向量检索找相近语义，用人工和模型复核判断能否引用。
- 对 OCR、人物档案、老报纸等高风险资料保留“候选、待核、可引用”的清晰边界。
- 通过 GitHub 固化 Codex 与 MiniMax 的交接成果。

## 分工原则

- Codex / GPT-5.5：唯一主控，负责方向、架构、关键代码、风险判断、配置变更、结果审核和最终合并。
- MiniMax：执行型助手，承担低风险、批量、重复、可验证任务，包括批量阅读、报告整理、抽样复核、测试补充和候选草稿。
- 本地模型：承担 OCR、向量化、批量转换等消耗型任务。
- GitHub：作为交接台，保存可公开的工程控制面、任务队列、审核清单和代码改动。

MiniMax 只能推送 `minimax/*` 分支，不允许 push main、merge main、删除原始资料、覆盖生产数据或改核心配置。

- 上海盟讯 OCR 已完成全文层处理，但仍属于 `ocr_unverified`，不能直接作为引用定稿。

- 4B reranker 暂缓，等资料库稳定和评测题体系成型后再决定。

## 公开边界

本仓库可以保存：

- README、PRD、SOP、任务计划和交接清单。
- 脚本模板、配置样例和不含私密路径/全文的公开小样本。
- Codex / MiniMax 的审核流程和候选规则说明。

本仓库不保存：

- 原始工作资料。
- 内部全文、OCR 全文、人物档案全文。
- 向量文件、索引文件、SQLite 数据库。
- 密钥、API key、私人路径、未脱敏报告。
- 未经核验却会被误认为定稿的材料。

## 交接文件

- `docs/control-plane/TASK_QUEUE_MINIMAX.md`：Codex 额度不足时，MiniMax 的接力任务队列。
- `docs/control-plane/CODEX_REVIEW.md`：Codex 恢复后最低成本审核清单。
- `docs/control-plane/PUBLICATION_BOUNDARY.md`：公开仓库发布边界。

## 本地使用原则

正式知识库运行在本地目录中，本仓库只发布工程控制面。任何全量 OCR、向量化、索引重建、原始资料整理，都应在本地知识库目录完成，并在通过脱敏和 Codex 审核后，只把必要的流程文件或公开样例同步到 GitHub。
